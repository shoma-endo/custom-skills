# E2Eテストケース設計スキル

対象リポジトリを解析し、機能要件・非機能要件のテストケース表 `docs/TEST_CASES.md` を生成する。読者は「これを見ながら手動テストを実行する自分（または引き継いだ人）」。自動化はスコープ外 — まず実行可能な設計書を作る。

## 実行方針（確認プロンプトを発生させない）

このスキルは `allowed-tools`（Read / Write / Edit / 下記の narrow な Bash 許可リスト）だけで完走できるように設計されている。ユーザー確認が挟まると自動走行が止まるため、以下を徹底する:

- **Bashコマンドは1コマンド1呼び出し**。`&&` `;` `|` で他コマンドと連結しない（連結すると許可パターンにマッチせず確認が発生しうる）
- テキスト加工（ケース表の組み立て・報告文の生成）は **Read の内容を自分で組み立てて Write/Edit する**。`python3` `node` 等の外部インタプリタを呼び出さない
- Step 2 の機能インベントリ提示は**ユーザーへの内容確認**であり、これは維持する（対象範囲を決める設計判断のため）。スキップ対象はあくまでツール実行時の**許可プロンプト**であって、この確認ステップではない

## Step 1: プロジェクト理解（推測禁止）

1. **`docs/PROJECT_OVERVIEW.md` があれば最優先で読む**（project-catchupの成果物。機能・構成・スタックの正本）。無ければ生成を提案しつつ、自力解析で続行する
2. README → CLAUDE.md / AGENTS.md → docs/ 配下
3. マニフェスト自動検出（Node決め打ちにしない）: `package.json` / `pnpm-workspace.yaml`（monorepoなら apps/* を列挙） / `requirements.txt` / `go.mod` / `appsscript.json` 等
4. **機能の実体を列挙**: ルーティング（Next.js `app/`・`pages/`・API routes）、CLIコマンド、定期ジョブ、Webhook受信
5. **データとリスクの所在**: スキーマ（`prisma/schema.prisma` 等）、認証・認可（next-auth・middleware・Lark scope）、外部連携（`.env.example` のキー名から。**実envは読まない**）
6. 既存テスト資産: vitest / playwright / cypress の有無 → 「自動化済み」「手動」の初期値に使う
7. **解析中に見つけた不具合は捨てない**: バグの疑い・仕様と実装の矛盾・ドキュメントとコードの不一致・未保護に見えるエンドポイント等は、その場で発見事項として控える。**疑いのままにせず、コードを読んで裏取りしてから確度を確定する**（確認の副作用コマンドは不可・読むだけ）

## Step 2: 機能インベントリの提示（1回だけ確認）

ユーザーフロー単位の機能一覧＋想定優先度を表で提示し、対象範囲の修正を受け付ける。「このままでいい」または応答が実質ない場合は提示内容のまま進む（止まらない）。

優先度基準:

| 優先度 | 基準 |
|--------|------|
| P0 | 壊れると業務が止まる/データが壊れる: 認証・認可、データ書き込み、課金、主要フローの正常系 |
| P1 | 主要機能の異常系・境界値、外部連携の失敗時挙動 |
| P2 | エッジケース、表示崩れ、利便性 |

## Step 3: 機能要件テストケース生成

フロー単位で **正常系・異常系・境界値** を設計する。各ケースの必須フィールド:

- ID（`TC-<フロー略号>-<連番>`）／優先度／前提条件／手順（番号付き・具体値）／期待結果／実行方法（手動 or 自動化済み: テストファイルパス）／根拠（出典: ルート・スキーマ・README のどこから導いたか）

境界値の定番（該当するものだけ使う）:
- ファイル/CSV処理 → 空ファイル・巨大ファイル・文字コード（UTF-8 BOM/Shift_JIS）・改行コード・カンマ/クォートのエスケープ・ヘッダ欠落・列数不一致
- フォーム → 空・最大長・絵文字/サロゲートペア・前後空白・SQLに見える文字列
- 一覧・ページング → 0件・1件・境界ページ・同時更新後の表示
- 権限 → **他ユーザーのリソースIDを直接指定した横アクセス**・ログアウト状態での直リンク・セッション失効後の操作

## Step 4: 非機能要件テストケース生成

**検出したスタックに該当するセクションだけ**生成する（全部盛りにしない）:

| 検出物 | 生成する観点 |
|--------|-------------|
| Vercel / Next.js | 応答時間・コールドスタート・ビルド成功・環境変数欠落時の起動挙動 |
| Prisma / DB | 同時更新の整合性・トランザクション境界・マイグレーション後方互換・バックアップからの復元手順 |
| next-auth / 認証 | セッション固定・失効・CSRF・認可漏れ（Step 3の横アクセスと重複させず参照） |
| LLM API（Anthropic等） | 不正JSON応答の後処理ガード・タイムアウト・レート制限時のリトライ・プロンプトインジェクション耐性 |
| Lark連携 | 権限スコープ不足時のエラー・テナント分離・token失効時の再取得・APIレート制限 |
| UIがある | 主要ブラウザ・モバイル幅・キーボード操作/フォーカス（アクセシビリティ基礎） |

形式は「チェック項目＋測定方法＋合格基準」。**根拠のない数値目標を作らない** — 性能目標等が未定義なら `目標値TBD（要ユーザー決定）` と書く。

## Step 5: 出力 `docs/TEST_CASES.md`

```markdown
---
generated: YYYY-MM-DD
commit: <HEAD SHA短縮形>
generator: e2e-testcases
---

# <プロジェクト名> E2Eテストケース

## 30秒サマリー（対象範囲・ケース数・P0数・前提環境）
## 機能テストケース（フロー別の表）
## 非機能テストケース（該当スタック分のみ）
## 発見事項（バグ・要確認 — 解析中に見つけたもの。0件なら「なし」と書く）
## トレーサビリティ（機能インベントリ → ケースID対応表）
## 実行記録（実施日 / 実施者 / 対象commit / 結果サマリー — 手動追記欄）
```

発見事項の必須フィールド: 確度（**確認済み**=コードで裏取り済み / **疑い**=未裏取り）／深刻度（高=データ破壊・セキュリティ / 中=機能不全 / 低=ドキュメント不一致等）／証拠（`file:line`）／内容と影響。**修正はしない**（このスキルは設計と報告まで。修正はユーザーの指示で別途）。解消済みの疑い（調べたら問題なかったもの）も「確認済み・問題なし」として残す — 次回の再調査を防ぐ。

- 日本語のみ。事実に出典を付け、不明は `TBD` と書く
- **既存の `docs/TEST_CASES.md` がある場合、「実行記録」と手動追記された結果は絶対に消さない**。ケース表は差分更新し、廃止したケースは削除でなく `廃止（理由）` とマークする

## Step 6: 報告

**発見事項を最初に報告する**（深刻度順。バグはテストケースより価値が高い）。続けてケース数（P0/P1/P2内訳）・非機能で生成したセクション・TBD一覧・推奨する次アクション（P0の手動実行、または自動化導入の判断材料）。

## 安全ルール

- 書き込みは対象リポジトリの `docs/TEST_CASES.md` のみ。コード・設定は変更しない
- テストは実行しない。dev server・ビルド・マイグレーション等の副作用コマンドも実行しない（記載のみ）
- `.env*`（sample以外）・credentials・秘密鍵は読まない
- コミットはしない（ユーザーの指示があれば行う）
