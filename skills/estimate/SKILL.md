# 開発案件見積もりスキル

案件情報から内部計算ノートと顧客向け見積書の2点を生成する。**単価・係数は `rate-card.yaml` が唯一の正本** — 本ファイルに数値を書かない。金額を出す前に必ず Read する。

| 成果物 | 保存先 | 役割 |
|--------|--------|------|
| ヒアリングシート | `~/Obsidian Vault/10-プロジェクト/見積もり/YYYY-MM-DD-<案件名>-ヒアリング.md` | 見積もり前に必ず作る質問リスト。顧客向けセクションはそのまま送れる文面 |
| 内部計算ノート | 同フォルダ `YYYY-MM-DD-<案件名>-内部.md` | 単価・工数・係数が見える計算過程＋実績記録欄。**校正資産** |
| 見積書 | 同フォルダ `YYYY-MM-DD-<案件名>-見積書.md` | 顧客向け。成果物単位の固定額のみ。**時給・係数は絶対に載せない** |

## Step 1: 質問シート生成（見積もりの前に必ず挟む）

案件の初期情報を受けたら、**いきなり見積もらない**。まず `templates/ヒアリング.md` に沿って質問シートを生成する:

1. 全案件共通の固定質問に加え、**この案件特有の質問**を初期情報の穴から導出する。質問は**業務・現状・体制に限る** — 技術的な質問（認証方式・スタック・設計方式など）は顧客にしない。技術はこちらの裁量で決め、決めた前提は見積書の「前提条件」に書いて提示する
2. 「顧客に聞く」（送れる文面）と「自分で判定する」（クラス・技術方針・不確実性・顧問余地）を分ける
3. シートを Vault に保存し、ユーザーに提示して回答を待つ。**この時点では見積もりに進まない**
4. 回答が揃ったら、またはユーザーが「これで見積もって」と言ったら Step 2 へ。未回答項目は仮置きし「仮」と明記して完走する
5. クラス判定に迷う案件は light / full 両クラスで併記提示する

## Step 2: 工数算出

1. `rate-card.yaml` を Read する
2. 成果物を機能単位に分解し、**アンカー工程**（light=実装 / full=開発）の時間を機能ごとに積み上げる。各見積もりに根拠1行（類似実績・複雑度）を付ける
3. アンカー工数 × 各工程の `effort_ratio` で全工程に展開する
4. 合計に `pm_overhead` を加算する
5. `risk_buffer` の該当係数を掛ける（どのクラスをなぜ選んだか内部ノートに明記）

該当しない工程（例: ライト案件に設計書不要）は削ってよいが、削った事実と理由を内部ノートに残す。

## Step 3: 金額算出

- 工程ごとに `時間 × base_rate × rate_factor` で内部金額を出す
- 顧客提示は**成果物・工程グループ単位の固定額**に組み替え、`pricing.rounding` 単位で切り上げる
- 見積書には**小計（税別）→消費税（`pricing.tax_rate`）→合計（税込）を並記**し、御見積金額の枠は税込額を出す
- 有効期限・支払条件・無償修正回数は rate-card の `pricing` に従う

## Step 4: スケジュール算出

- 総工数 ÷ `effective_hours_per_day` ÷ `work_days_per_week` で自分の作業週数を出す（専念案件なら稼働を都度上書き）
- 承認ゲート（要件承認・設計承認・検収）ごとに `client_review_days` をカレンダーに挿入する。**顧客レビューが実際のボトルネック** — マイルストーン表に「顧客確認期間」を明示し、遅延時は納期が後ろ倒しになる旨を見積書に書く

## Step 5: マーケ要素の分離

開発見積もりにレベシェアを混ぜない。マーケ要素がある場合:

- 開発 = 固定額（本見積書）
- マーケ支援 = 別契約として概要のみ添える: `marketing.min_monthly_fee` 以上の最低固定費＋**計測可能な指標**（何の売上・何%・計測方法・期間）へのレベシェア
- ファネル整理は見積書でなく提案書の要素。求められたら別ファイルで出す

## 開発顧問（月額契約）の扱い

顧問の依頼・提案余地がある場合は rate-card の `advisory` に従う:

- 月額 `monthly_fee`（税別）・稼働目安 `hours_guide_per_month`・含む/含まないは rate-card の値をそのまま使う
- **実装作業は顧問に含めない**。発生したらクラス単価で別途見積もり（このスキルの通常フロー）
- 開発見積書に混ぜず、見積書末尾にオプション1行で添えるか、顧問単独なら契約概要（月額・稼働目安・範囲・月次更新）を独立ファイルで出す
- 稼働超過が常態化したら増額改定の提案をユーザーに促す

## Step 6: 生成と報告

1. `templates/内部計算.md` に沿って内部ノートを生成（frontmatter の status: draft）
2. `templates/見積書.md` に沿って見積書MD（正本）を生成。**内部専用情報（時給・係数・バッファ率）が混入していないか生成後にセルフチェック**
3. **送付用PDFを生成**: `templates/見積書.html` のプレースホルダ（`{{CLIENT_NAME}}` 等）にMD正本の内容を流し込み、同フォルダに一時HTMLを書いてから Chrome headless でPDF化:

   ```bash
   "/Applications/Google Chrome.app/Contents/MacOS/Google Chrome" --headless --disable-gpu \
     --no-pdf-header-footer --print-to-pdf="<出力先>.pdf" "file://<一時HTML絶対パス>"
   ```

   - 発行者情報は rate-card の `issuer` から。address が空なら行ごと省略
   - 生成後にPDFを Read して崩れ（文字化け・表のはみ出し・ページ割れ）がないか目視確認し、一時HTMLは削除。PDF生成に失敗したらMDを正として報告し完走する（止まらない）
   - 修正依頼が来たら**MD正本を直してPDFを再生成**する。PDFを直接編集しない
4. 報告: ファイルパス（MD2点＋PDF）・提示合計額・想定納期・仮置きした項目の一覧

## 実績記録モード（「見積もり実績を記録して」）

案件完了時に呼ばれたら:

1. 該当の内部ノートを開き「見積 vs 実績」表に実績時間を追記、status を done に更新
2. 工程別の乖離率を算出。**±30% を超える工程があれば rate-card.yaml の該当係数（effort_ratio / risk_buffer / effective_hours_per_day）の修正案を提示**する（勝手に書き換えない。承認後に Edit）
3. 学び1〜3行をノート末尾に記録

数件溜まったこの記録が係数校正の唯一の根拠になる。推測で係数を動かさない。

## 安全ルール

- 書き込みは Vault の `10-プロジェクト/見積もり/` 配下と（承認後の）`rate-card.yaml` のみ
- 見積書・内部ノートとも日本語のみで書く
- 顧客名・金額を含むファイルを Vault 外にコピーしない。見積書の送付はユーザーが行う（このスキルは送信しない）
- 過去の見積もりノートは削除・上書きしない（同名衝突時は `-2` を付ける）
