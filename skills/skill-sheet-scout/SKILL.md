# skill-sheet-scout — スキルシート月次更新の調査・提案

## 前提: 個人設定を先に読む

**開始時に必ず同ディレクトリの `local-config.md` を Read する**（gitignore対象・リポジトリには含まれない）。シートのURL/シート名/レンジ、リポジトリ→案件マッピング、参照パス、ctx検索クエリはすべてそこにある。無ければユーザーに作成を依頼して停止する。以下の手順で「（local-config参照）」とある箇所は同ファイルの値を使う。

## 目的とステークス

スキルシート（技術経歴書）は **SES営業担当とクライアント先に提出される営業資料**。ここで単価と任せられるポジションが決まる。単なる作業ログではなく「単価を上げる根拠」を書く場所として扱う。

- シート構成: ①SES経歴（項番1〜N）②スキル要約（自己PR）③保有技術マトリクス（工程/言語/FW、A〜C評価）④受託開発セクション
- **このskillは提案専用。シートへの転記は本人が手動で行う（2026-07-20 本人指示）。** lark-sheets での書き込みは提案しない

## 単価・ポジションに効く5観点（分析の軸）

1. **AI駆動開発の実運用** — LLM API組込機能、spec-to-PR等のワークフロー運用、AI活用の定量効果。2026年時点で最大の差別化
2. **リーダー / PM / 顧客折衝** — 定例MTG主導、スケジュール調整、コスト管理、レビュワー経験
3. **上流・アドバイザリー** — 要件定義、セキュリティ/UI監修、技術顧問。作業者から設計レイヤーへの証明
4. **定量成果** — パフォーマンス改善の数値、規模、期間。「約20秒→約5秒」のように具体数字で
5. **技術以外の強み** — 講師・登壇（定例講座、勉強会）、OSS貢献、技術発信（note/zenn/X）、外注管理

## 収集手順

期間 = 実行日の1ヶ月前〜当日。`SINCE=$(date -v-1m +%Y-%m-%d)` で統一する。

### 1. git 全リポジトリ掃引

```bash
for d in ~/dev/*/; do if [ -d "$d/.git" ]; then n=$(git -C "$d" log --since=$SINCE --oneline 2>/dev/null | wc -l | tr -d ' '); if [ "$n" != "0" ]; then echo "=== $d ($n commits)"; git -C "$d" log --since=$SINCE --pretty='%ad %s' --date=short | head -20; fi; fi; done
```

### 2. リポジトリ→案件マッピング（local-config参照・推測断定禁止）

確定対応のみ使う。未確定リポジトリは「要確認」として列挙し、勝手にマッピングしない。ローカルにコードが無い案件（SES先環境で完結する本業等）は、業務内容をヒアリング項目として質問リスト化する。

### 3. ノート・契約・発信（local-config参照）

直近更新ノート、契約ログ（案件の層・役割の正本）、長文発信の下書き・公開実績を確認する。

### 4. Lark 行動履歴（MTG・講師・折衝の証跡）

```bash
lark-cli calendar +agenda --start "${SINCE}T00:00:00+09:00" --end "$(date +%Y-%m-%d)T23:59:59+09:00" --format json --as user | jq -r '.data[] | .summary' | sort | uniq -c | sort -rn
lark-cli vc +search --start "$SINCE" --end "$(date +%Y-%m-%d)" --as user
```

定例講座の回数、案件定例、実装相談・顧客折衝を数える。

### 5. コーディングエージェント履歴の横断検索（ctx）

git に残らない作業（調査・設計相談・レビュー・アドバイザリー対応）を拾う。ctx-agent-history-search skill 併用。`ctx status` で索引済みを確認してから:

```bash
ctx search "<キーワード>" --since 30d
```

最低限回すクエリ:
- **案件名**（local-config のクエリリスト）。ローカルにコードが無い案件は、エージェント履歴が唯一の証跡になり得る
- **作業種別**: 「設計」「レビュー」「調査」「セキュリティ」等、git コミットに対応しない大きなセッション

ヒットしたら `ctx show event <id> --window 5` / `ctx show session <id>` で内容確認してから証跡に使う。憶測で業務内容に昇格させない（事実と解釈の分離）。

### 6. シート現状の読み取り（local-config参照）

lark-sheets skill（`lark-cli sheets +cells-get`）でシート現状を読む。ブラウザ（claude-in-chrome）はフォールバック。見るポイント:

- 最新SES項番の業務内容・期間セル（継続中表記が古くなっていないか）
- スキル要約（自己PR）が現在のスタックとズレていないか
- 保有技術マトリクスに直近スタックの行があるか
- 受託開発セクションの実務経験欄と、未記載の稼働中案件

## 出力形式

シートの記法に合わせる: **社名は書かない**（「業界特化のマーケティングSaaS」「顧客サポートプラットフォーム」式のぼかし）。事実（コミット・イベント）と解釈（アピール文言）を分ける。

1. **追記候補（優先度順・セクション別）** — そのまま転記できる文面案＋根拠（リポジトリ/コミット/カレンダーイベント）
2. **新規項目候補** — 未記載の稼働案件（期間・概要・使用技術・役割・規模のセル構成で下書き）
3. **自己PR・保有技術の更新案** — 現況とのズレ指摘
4. **要確認リスト** — 未確定マッピング、ヒアリング質問、期間セル更新

提案を出したら終了。転記の申し出はしない。

## 注意

- 盛らない。コミット・イベントの証跡が無いものは書かない（面談で突っ込まれて崩れる）
- 数字で書けるものは数字で（改善前後、規模、回数、期間）
- 誇張表現は不可。事実と解釈の分離、基準値で判断
- **このSKILL.mdは公開リポジトリ管理。クライアント名・URL・個人パス等の機微情報は必ず local-config.md 側に書く**
